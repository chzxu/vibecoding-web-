🃏 CTF 题型速查

第一次接触 CTF 的菜鸡必看！搞懂每种题型是干什么的，才知道从哪开始下手。

🌐
Web
简单 😊 新手首选

网页漏洞找 Flag。考 SQL 注入、XSS、文件上传、SSRF 等。**最推荐新手入门方向。**

Burp Suite · Chrome F12 · SQLMap
📦
Misc
简单~中等 😊 捡漏机会多

杂项，啥都可能考：图片隐写、流量分析、编码解码、压缩包密码… 考的是脑洞和细心。

StegSolve · Wireshark · binwalk
🔐
Crypto
中等 🤔 看数学天赋

密码学题目。古典密码（凯撒、维吉尼亚）、现代密码（RSA、AES）、哈希碰撞。要懂一点数学基础。

Python · RsaCtfTool · CyberChef
🔍
Reverse
较难 😰 要有耐心

逆向工程。给你一个程序，让你分析它的逻辑、找到 Flag。要会看汇编代码、脱壳、反混淆。

IDA Pro · Ghidra · x64dbg
💥
Pwn
困难 😱 大佬向

二进制漏洞利用。栈溢出、堆攻击、格式化字符串… 要精通 C/汇编和系统原理。难度天花板。

pwntools · gdb · one_gadget
🎯 给菜鸡的第一条建议

先死磕 Web！Web 题学到的知识（HTTP 协议、SQL 语句、JS 代码）在现实中也能用到，成就感来得快，适合建立信心。Web 玩溜了再碰 Misc 和 Crypto，最后再去 Reverse 和 Pwn。

📚 Web技术演化

摘自《Web安全学习笔记》第1章 · 了解Web的前世今生，打好安全基础。

▶

🌐 简单网站

Web起源

Web技术的最初阶段，网站内容基本都是静态的，由文字和图片组成，用户行为也简单——基本只是浏览网页。

📄 静态页面

早期网站托管在 ISP 上，用表格布局，纯展示内容。

🎬 多媒体阶段

音频、视频、Flash 技术加入，网页变得生动形象。

⚙️ CGI 阶段

CGI（Common Gateway Interface）定义了Web服务器与外部程序的通信标准。PHP/ASP/JSP 等语言入场，实现动态内容生成。

🏗️ MVC 架构

Model-View-Controller 分离关注点，代码复用性更好。还出现了 MVP、MVVM 等变体，以及 ORM 概念替代直接 SQL 操作。

▶

🔗 数据交互

前后端通信
📋 简单数据交互

早期使用 Web 表单、XML、SOAP 等方式传递数据。

⚡ Ajax

Asynchronous Javascript And XML —— 无需刷新整个页面就能更新数据，用户体验质的飞跃。

🔌 RESTful

基于 URL 组织功能，充分利用 HTTP 语义。关键特征：

  • 域名和主域名分开（api.example.com）
  • 带版本控制（/v1/ /v2/）
  • 用 HTTP 动词描述操作（GET/POST/PUT/DELETE）
  • 正确使用状态码 + JSON 响应格式
🕸️ GraphQL

Facebook 推出的图状数据查询方式，适合复杂依赖关系的场景。

💾 二进制

HTTP/2、Protocol Buffer 等高效率方式加速数据交互。

▶

🏛️ 架构演进

系统复杂度

业务发展推动架构越来越复杂，功能被拆分成不同模块。客户端也从 Web 扩展到 iOS、Android 等多端。

数据库方面，除 MySQL/PostgreSQL 外，出现了 Redis 缓存、Elasticsearch 搜索、Kafka 消息队列等专业数据组件。

🔧 中间件

用于管理计算资源和网络通信的独立程序。常见功能：过滤IP、合并端口、路由、权限校验、负载均衡、反向代理等。

🌍 分布式

单台设备无法承载海量访问，功能也被拆分到不同应用，于是出现了分布式服务框架（RPC）。

▶

☁️ 云服务

云计算时代

从"裸金属"物理机到云主机（IaaS），再到容器化、微服务、Serverless，基础设施成本大幅降低。

⚡ Serverless

FaaS（函数即服务）+ BaaS（后端即服务）。用户只需上传逻辑函数，配置触发条件，云平台自动执行。

🧩 微服务

2014年由 Martin Fowler 提出。每个服务运行在自己的进程中，通过 HTTP API 通信，独立部署、版本控制、容量伸缩。

🚪 API网关

统一入口代理后端服务访问，服务治理特性对客户端透明，方便升级维护。

▶

🛠️ 软件开发

工程实践
🔄 CI/CD

持续集成（CI）：开发人员频繁将工作集成到共享分支，减少隔离和合并冲突。

持续交付（CD）：将构建从测试套件部署到类生产环境，让开发人员在不增加工作量的情况下验证 Bug 修复或新特性。

文件操作

8 条指令
ls
列出目录内容 —— 最常用的命令之一
📖 语法
ls [选项] [路径]
🔧 常用选项
-l 长格式显示（权限、大小、日期）
-a 显示所有文件（包括 . 开头的隐藏文件）
-h 人类可读大小（配合 -l 使用）
-t 按时间排序
-S 按大小排序
💡 示例
# 详细列出当前目录
$ ls -lah
drwxr-xr-x 5 user user 4.0K May 12 10:00 .
-rw-r--r-- 1 user user 256 May 12 09:30 notes.txt

# 按时间倒序
$ ls -lt
💡 ls -la 是你的"开盲盒"命令 —— 想看看目录里有什么，就用它！
cd
切换当前工作目录
📖 语法
cd [路径]
🔧 常用用法
cd ~ 回到 home 目录
cd / 回到根目录
cd .. 返回上一级目录
cd - 返回上一个目录
💡 示例
$ cd /var/log
$ cd ..
$ cd ~/Projects/my-app
💡 cd - 太实用了 — 在最后两个目录之间跳转，做运维的都知道！
pwd
显示当前工作目录的绝对路径
📖 语法
pwd
💡 示例
$ pwd
/home/zixu/projects/linux-course
💡 迷路时第一个想到的命令！pwd = print working directory。
mkdir
创建新目录
📖 语法
mkdir [选项] 目录名
🔧 常用选项
-p 递归创建父目录（不存在则自动创建）
-v 显示创建过程
💡 示例
# 创建单层目录
$ mkdir myproject

# 递归创建多层目录（不用一层层建了！）
$ mkdir -p projects/2026/05/笔记
💡 -p 参数是神技 — 建多层目录只用一条命令。
rm
删除文件或目录（⚠️ 危险！不可恢复）
📖 语法
rm [选项] 目标
🔧 常用选项
-r 递归删除（删除目录必加）
-f 强制删除，不提示
-rf ⚠️ 递归强制删除（慎用！）
💡 示例
# 删除文件
$ rm old-file.txt

# 删除目录（含所有内容）
$ rm -rf temp-folder
⚠️ rm -rf / 是最著名的"删库跑路"命令 —— 千万别乱试！（没有 -f 的话会一路问你确认）
cp
复制文件或目录
📖 语法
cp [选项] 来源 目标
🔧 常用选项
-r 递归复制（复制目录必加）
-i 覆盖前询问
-v 显示详细过程
-u 只复制更新的文件
💡 示例
# 复制文件
$ cp report.txt backup/

# 复制整个目录
$ cp -r my-project/ my-project-backup/
mv
移动文件 / 重命名文件
📖 语法
mv [选项] 来源 目标
🔧 常用选项
-i 覆盖前询问
-v 显示过程
💡 示例
# 重命名
$ mv old-name.txt new-name.txt

# 移动到其他目录
$ mv report.pdf ~/Documents/
💡 mv 即 rename — Linux 没有专门的重命名命令，mv 就是！
touch
创建空文件 / 更新文件时间戳
📖 语法
touch 文件名
💡 示例
# 创建空文件
$ touch notes.md index.html app.py

# 一次创建多个文件
$ touch {a,b,c}.txt → 创建 a.txt b.txt c.txt
💡 文件已存在时 touch 不会覆盖内容，只更新时间戳。超安全！

权限管理

5 条指令
chmod
修改文件或目录的权限
📖 语法
chmod [权限] 文件
🔧 权限表示法
数字法：chmod 755 → rwxr-xr-x
  r=4, w=2, x=1  所有者|组|其他人
符号法：chmod u+x → 给所有者加执行权限
💡 示例
# 数字法：所有者全部权限，组和其他只读+执行
$ chmod 755 script.sh

# 符号法：给所有人加执行权限
$ chmod +x run.sh
💡 755 = 脚本标准权限 | 644 = 普通文件标准权限 | 600 = 私密文件
chown
修改文件所有者（需 root）
📖 语法
chown [用户]:[组] 文件
💡 示例
# 修改所有者
$ sudo chown zixu config.yml

# 同时修改所有者和组
$ sudo chown zixu:developers app/
💡 用 -R 递归修改目录内所有文件：chown -R user:group dir/
chgrp
修改文件所属组
📖 语法
chgrp 组名 文件
💡 示例
$ sudo chgrp developers config.yml
$ chgrp -R www-data /var/www/
umask
设置新建文件的默认权限掩码
📖 语法
umask [掩码值]
💡 示例
# 查看当前 umask
$ umask
0022

# 设置新建文件默认为 755（目录）
$ umask 022
💡 umask = 777 - 目标权限。默认 022 意味着文件 644、目录 755。
sudo
以超级用户权限执行命令
📖 语法
sudo 命令
🔧 相关用法
sudo -i 切换到 root shell
sudo -u user 以指定用户身份执行
💡 示例
$ sudo apt update
$ sudo systemctl restart nginx
[sudo] password for zixu:
⚠️ 不是 root 用户！是"借权限执行一条命令"。sudo !! 用上一条命令加 sudo。

进程管理

6 条指令
ps
显示当前进程快照
📖 语法
ps [选项]
🔧 常用选项
ps aux 显示所有进程（BSD 风格）
ps -ef 显示所有进程（标准风格）
ps -u user 查看指定用户的进程
💡 示例
$ ps aux | grep nginx
root 12345 0.0 0.1 12345 6789 ? Ss 10:00 0:00 nginx: master
www-data 12346 0.0 0.1 12345 6789 ? S 10:00 0:00 nginx: worker
💡 ps aux | grep <进程名> — 查找进程的经典组合拳。
top
实时查看系统进程和资源占用
📖 语法
top
🔧 交互快捷键
q 退出
k 杀死进程（输入 PID）
M 按内存排序
P 按 CPU 排序
1 显示每个 CPU 核心
💡 top -u zixu 只查看你的进程。htop 是 top 的增强版（更漂亮），需要 apt install。
kill
发送信号终止进程
📖 语法
kill [信号] PID
🔧 常用信号
kill -15 SIGTERM — 礼貌请求退出（默认）
kill -9 SIGKILL — 强制杀死（⚠️ 无法被捕获）
kill -1 SIGHUP — 重新加载配置
💡 示例
# 先找到 PID
$ ps aux | grep firefox
zixu 23456 ... firefox
# 先试试礼貌退出
$ kill 23456
# 不行？上大招
$ kill -9 23456
💡 先 kill -15（给人家保存机会），不行再 kill -9。先礼后兵！
bg / fg
将进程放到后台/前台运行
📖 语法
command & 启动时直接放后台
Ctrl + Z 暂停当前进程
bg 将暂停的进程放到后台
fg 将后台进程调到前台

git remote 远程

📖 语法
git pull [远程名] [分支名]
💡 示例
# 拉取并合并 $ git pull # 拉取后变基（rebase）而不是合并，历史更干净 $ git pull --rebase # 首次拉取远程分支 $ git pull origin develop
💡推荐配置 git pull --rebase 替代默认的 merge 拉取，避免产生多余的合并节点。

💡 示例

💡git fetch 是安全的：只下载数据，不改动你的工作区。先 fetch 看看再决定如何合并。
git push --delete 删除
删除远程分支或标签
💡 示例

撤销 / 回退

5 条
git reset 回退
撤销提交、取消暂存或回退到指定版本
📖 语法
git reset [模式] [目标]
🔧 三种模式
--soft 只移动 HEAD，代码不变，暂存区不变 --mixed 移动 HEAD，代码不变，清除暂存（默认） --hard 移动 HEAD，代码也回退到目标版本（⚠️ 丢失修改）
💡 示例
# 撤销上一次提交（保留代码，重新提交） $ git reset --soft HEAD~1 # 取消暂存区的文件（= 旧版 git reset HEAD file） $ git reset index.html # 回退到指定提交（⚠️ 丢弃之后所有修改） $ git reset --hard 1a2b3c4
⚠️--hard 会丢失未提交的修改！如果已经 push 到远程，用 git revert 代替。
git revert 撤销
通过创建一个"反提交"来撤销之前的修改（安全！）
📖 语法
git revert 提交ID
💡 示例
# 撤销最近一次提交 $ git revert HEAD # 会创建一个新的提交，内容是对那次提交的反向操作 # 撤销指定范围的提交 $ git revert HEAD~3..HEAD # 不自动提交（方便修改提交信息） $ git revert --no-commit 1a2b3c4
💡多人协作用 revert 而不是 reset！revert 是安全的"反悔"，不影响他人历史。
git clean 清理
删除工作区中未被跟踪的文件和目录
📖 语法
git clean [选项]
💡 示例
# 预览待删除的文件 $ git clean -n # 删除未跟踪的文件 $ git clean -f # 删除未跟踪的目录 $ git clean -fd # 连 .gitignore 里的文件也删掉 $ git clean -xdf
⚠️先 git clean -n 预览，确认无误再加 -f。误删就找不回来了！
git reflog 救命
查看 HEAD 的所有历史移动记录（最后的救命稻草）
📖 语法
git reflog
💡 示例
# 查看所有操作历史 $ git reflog 1a2b3c4 (HEAD -> main) HEAD@{0}: reset: moving to HEAD~1 5e6f7g8 HEAD@{1}: commit: feat: 新增导航栏 9a0b1c2 HEAD@{2}: commit: fix: 首页样式 # 恢复到误操作之前的状态 $ git reset --hard HEAD@{1}
💡git reflog 是后悔药！不管怎么 reset、rebase 搞错了，只要 reflog 里还有记录就能恢复。

⚠️只适合修改还没有推送的提交。已 push 的别改——强行推送会害队友！

git rebase 变基
将当前分支的提交移动到另一个分支顶端，使历史更线性干净
📖 语法
git rebase 目标分支
💡 示例
# 把 feature 分支的提交移到 main 顶端 $ git switch feature $ git rebase main # 交互式变基（合并/修改提交历史） $ git rebase -i HEAD~3 # 把最近 3 个提交合为 1 个、修改顺序、更改信息等 # 交互式变基选项： pick 保留提交 squash 合并到上一个提交 reword 修改提交信息 edit 修改提交内容 drop 删除提交
⚠️已经推送到远程的公共分支 不要 rebase！会导致别人的本地和远程历史不一致。
rebase 只适合本地分支或单人开发分支。
git cherry-pick 挑拣
将某个或某些提交复制到当前分支
📖 语法
git cherry-pick 提交ID
💡 示例
# 复制单个提交 $ git cherry-pick 1a2b3c4 # 复制连续多个提交 $ git cherry-pick a1b2c3d..e5f6g7h # 只复制修改但不自动提交（方便修改） $ git cherry-pick -n 1a2b3c4
💡经典场景：hotfix 修好了 main 分支的 Bug，用 cherry-pick 复制到 release/ 分支，不用合并整个 main。

合并冲突时的标准处理流程
📖 冲突标记
# 出现冲突时文件内容长这样： <<<<<<< HEAD（当前分支的版本） console.log("当前分支代码"); ======= console.log("合并进来的代码"); >>>>>>> feature-branch（被合并的分支）
📖 解决流程
# 1. 查看哪些文件冲突 $ git status # 2. 手动编辑文件解决冲突（删掉 <<< === >>> 标记） # 3. 标记已解决 $ git add resolved-file.js # 4. 完成合并 $ git commit
💡git merge --abort 可以取消当前合并，回到开始前的状态。先 abort 冷静一下再重来！

PR 合并到 main 后，本地删除分支 $ git switch main $ git pull $ git branch -d feature-my-work

git tag 标签
给某个提交打标签（通常用于版本发布）
📖 语法
git tag [选项] 标签名
💡 示例
# 创建轻量标签 $ git tag v1.0.0 # 创建附注标签（推荐，带作者和日期信息） $ git tag -a v1.0.0 -m "Release v1.0.0 - 登录功能完成" # 给历史提交打标签 $ git tag -a v0.9.0 1a2b3c4 -m "Beta 版" # 列出所有标签 $ git tag # 推送标签到远程 $ git push origin v1.0.0 # 推送所有标签 $ git push --tags

git bisect 二分
二分查找 —— 定位 Bug 是哪个提交引入的
📖 使用
# 启动 bisect $ git bisect start # 告诉 Git"这个提交是坏的"（当前 HEAD） $ git bisect bad # 告诉 Git"某个旧版本是好的" $ git bisect good v1.0.0 # Git 会自动切换到中间的提交。 # 测试这个版本，然后标记： $ git bisect good # 如果没问题 $ git bisect bad # 如果有问题 # 重复几次，Git 会精确到引入 Bug 的那个提交 # 结束 bisect $ git bisect reset
💡自动 bisect：git bisect run npm test —— 用测试脚本自动跑！适合回归测试定位问题。

git worktree add 路径 分支名

git submodule 子模块
在项目中引用其他 Git 仓库（依赖管理）
📖 使用
# 添加子模块 $ git submodule add https://github.com/user/lib.git lib/ # 克隆包含子模块的项目（首次） $ git clone --recurse-submodules https://github.com/user/project.git # 更新子模块到最新 $ git submodule update --remote # 如果忘了 --recurse-submodules $ git submodule update --init --recursive

git hooks 钩子
在特定 Git 操作前后自动执行脚本
📖 常用钩子
pre-commit 提交前运行（检查代码格式、跑 lint）
pre-push 推送前运行（跑测试）
post-merge 合并后运行（安装新依赖）
prepare-msg 编辑提交信息前运行
💡 示例
# .git/hooks/pre-commit 示例（防止提交 console.log） #!/bin/sh if git diff --cached | grep -E "console\.(log|debug)"; then echo "❌ 提交中包含 console.log，请先清理！" exit 1 fi
💡.git/hooks/ 里的钩子不会提交到仓库。团队统一管理的话用 husky（npm）或 pre-commit（Python）。
