基础操作

7 条
git init 初始化
在当前目录初始化一个新的 Git 仓库
📖 语法
git init
💡 示例
# 初始化当前目录 $ git init Initialized empty Git repository in /home/user/project/.git/ # 创建空仓库并初始化 $ git init my-project
💡一个项目只需要执行一次，会在项目根目录生成 .git/ 文件夹。
git clone 克隆
从远程仓库克隆代码到本地
📖 语法
git clone 仓库地址 [目录名]
💡 示例
# HTTPS 克隆 $ git clone https://github.com/user/repo.git # SSH 克隆 $ git clone git@github.com:user/repo.git # 指定目录名 $ git clone https://github.com/user/repo.git my-folder # 只克隆最近一次提交（速度快） $ git clone --depth 1 https://github.com/user/repo.git
💡SSH 方式需要提前配置公钥。--depth 1 适合大项目节省带宽和磁盘。
git add 暂存
将文件变更添加到暂存区（准备提交）
📖 语法
git add [文件/目录]
💡 示例
# 添加单个文件 $ git add index.html # 添加所有变更（最常用！） $ git add . # 或者 $ git add -A # 添加所有 .js 文件 $ git add *.js
💡git add -p 可以交互式地选择每个块是否添加，非常精细！
git commit 提交
将暂存区的变更提交到本地仓库
📖 语法
git commit [选项]
💡 示例
# 最常用：直接写提交信息 $ git commit -m "feat: 新增用户登录功能" # 跳过 add，直接提交已跟踪文件的修改 $ git commit -a -m "fix: 修复首页样式" # 编辑器模式（写多行提交信息） $ git commit # 会打开 vim 让你写详细的提交信息
💡好的提交信息格式：类型: 简短说明
如 feat: 新功能、fix: 修复、refactor: 重构、docs: 文档。
git status 状态
查看工作区和暂存区的当前状态
📖 语法
git status
💡 示例
# 简洁模式（推荐） $ git status -s M README.md # 已修改未暂存 A newfile.txt # 已暂存 ?? untracked.py # 未跟踪 # 完整模式 $ git status
💡最常用的命令之一！随时 git status 看看当前仓库状态，心里有底。
git log 日志
查看提交历史记录
📖 语法
git log [选项]
💡 示例
# 简洁的一行式日志 $ git log --oneline --graph --decorate * 1a2b3c4 (HEAD -> main) feat: 首页重构 * 5e6f7g8 fix: 修复登录 bug * 9a0b1c2 (origin/main) init: 项目初始化 # 显示最近 5 条 $ git log -5 # 查看某个文件的提交历史 $ git log --oneline index.html
💡别名推荐：git config --global alias.lg "log --oneline --graph --decorate --all"
之后只需 git lg 就能看清晰历史树！
git rm / mv 管理
删除或移动/重命名 Git 管理的文件
📖 语法
git rm 文件 # 删除文件并自动暂存 git mv 旧名 新名 # 重命名文件并自动暂存
💡 示例
# 删除文件（比直接删 + git add 省一步） $ git rm old-file.txt # 从 Git 跟踪中移除但不删除本地文件 $ git rm --cached config.local.js # 重命名 $ git mv README.md README_ZH.md
⚠️git rm --cached 可以"取消跟踪"文件，不删除本地副本。适合把不该提交的文件从 Git 移除。

分支管理

6 条
git branch 分支
创建、查看、删除分支
📖 语法
git branch [选项] [分支名]
💡 示例
# 列出本地分支（* 表示当前分支） $ git branch develop * main feature-login # 创建新分支 $ git branch feature-nav # 删除已合并的分支 $ git branch -d feature-old # 强制删除分支（无论是否合并） $ git branch -D feature-abandoned # 查看本地和远程所有分支 $ git branch -a
💡分支名规范：feature/xxx、bugfix/xxx、hotfix/xxx，团队统一风格。
git checkout 切换
切换分支 或 恢复文件
📖 语法
git checkout 分支名
💡 示例
# 切换到已有分支 $ git checkout develop # 创建并切换到新分支（两步合一） $ git checkout -b feature-search # 撤销对某个文件的修改（⚠️ 谨慎！会丢失工作区修改） $ git checkout -- index.html
💡git switch 是 Git 2.23+ 引入的新命令，专门用于切换分支。
git switch develop 和 git switch -c new-branch。语义更清晰。
git merge 合并
将指定分支的修改合并到当前分支
📖 语法
git merge 分支名
💡 示例
# 先把 feature 合并到 main $ git switch main $ git merge feature-login # 合并但不自动提交（方便检查冲突） $ git merge --no-commit feature # 提交时保留分支历史（一目了然） $ git merge --no-ff feature
💡合并时遇到冲突？文件里会有 <<<<<<< 标记，手动解决后 git add + git commit。
git switch 切换
Git 2.23+ 引入的专门切换分支命令
📖 语法
git switch 分支名 # 切换 git switch -c 新分支 # 创建并切换
💡 示例
$ git switch main $ git switch -c feature-header
💡比 git checkout 更安全 —— 不会意外覆盖工作区文件。
git restore 恢复
恢复工作区或暂存区的文件（Git 2.23+）
📖 语法
git restore 文件 # 恢复工作区文件（取消修改） git restore --staged # 取消暂存
💡 示例
# 恢复文件到最近一次提交的状态 $ git restore index.html # 取消暂存（= 旧版的 git reset HEAD） $ git restore --staged index.html
💡git restore 取代了 git checkout -- file 和 git reset HEAD file，语义更清楚。
📐 分支策略参考 规范
常见 Git 分支工作流
📖 Git Flow
main → 发布给用户的稳定版本 develop → 日常开发主分支 feature/xxx → 新功能从 develop 分出 release/x.x → 发布准备分支 hotfix/xxx → 紧急修复从 main 分出
📖 GitHub Flow（更简洁）
main → 永远可部署 feature/xxx → 从 main 分支 → PR 合并回 main
💡小团队推荐 GitHub Flow：一个 main 分支 + 功能分支 + Pull Request，简单高效。

远程仓库

6 条
git remote 远程
管理远程仓库的地址和别名
📖 语法
git remote [选项]
💡 示例
# 查看远程仓库列表 $ git remote -v origin https://github.com/user/repo.git (fetch) origin https://github.com/user/repo.git (push) # 添加远程仓库 $ git remote add upstream https://github.com/other/repo.git # 删除远程仓库 $ git remote remove old-remote # 修改远程仓库地址 $ git remote set-url origin git@github.com:user/repo.git
💡origin 是默认的远程仓库名，你也可以加多个，如 upstream（上游仓库）。
git push 推送
将本地提交推送到远程仓库
📖 语法
git push [远程名] [分支名]
💡 示例
# 推送到默认远程 origin 的当前分支 $ git push # 指定远程和分支 $ git push origin main # 推送并设置上游（首次推送新分支时） $ git push -u origin feature-login # 之后只需 git push # 强制推送（⚠️ 覆盖远程历史！） $ git push --force
⚠️永远不要在公共分支上 git push --force！会覆盖别人提交。
用 --force-with-lease 更安全（会检查是否有远程新提交）。
git pull 拉取
从远程仓库拉取最新代码并合并（= fetch + merge）
📖 语法
git pull [远程名] [分支名]
💡 示例
# 拉取并合并 $ git pull # 拉取后变基（rebase）而不是合并，历史更干净 $ git pull --rebase # 首次拉取远程分支 $ git pull origin develop
💡推荐配置 git pull --rebase 替代默认的 merge 拉取，避免产生多余的合并节点。
git config --global pull.rebase true
git fetch 获取
获取远程仓库的最新变更（不自动合并）
📖 语法
git fetch [远程名]
💡 示例
# 获取所有远程分支的最新信息 $ git fetch --all # 查看远程分支的更新，但不合并 $ git fetch origin $ git log --oneline origin/main
💡git fetch 是安全的：只下载数据，不改动你的工作区。先 fetch 看看再决定如何合并。
git push --delete 删除
删除远程分支或标签
💡 示例
# 删除远程分支 $ git push origin --delete feature-old # 或简写（推送空分支到远程 = 删除） $ git push origin :feature-old # 删除远程标签 $ git push origin --delete v1.0.0
git lfs 大文件
Git Large File Storage —— 管理大文件
📖 使用
# 安装（需要先安装 git-lfs） $ git lfs install # 跟踪大文件类型 $ git lfs track "*.psd" $ git lfs track "*.zip" # 提交 .gitattributes $ git add .gitattributes $ git commit -m "chore: add LFS for large files"
💡GitHub 免费版 LFS 限额 1GB，大文件（设计稿、数据集）用 LFS 代替 Git 直接存储。

撤销 / 回退

5 条
git reset 回退
撤销提交、取消暂存或回退到指定版本
📖 语法
git reset [模式] [目标]
🔧 三种模式
--soft 只移动 HEAD，代码不变，暂存区不变 --mixed 移动 HEAD，代码不变，清除暂存（默认） --hard 移动 HEAD，代码也回退到目标版本（⚠️ 丢失修改）
💡 示例
# 撤销上一次提交（保留代码，重新提交） $ git reset --soft HEAD~1 # 取消暂存区的文件（= 旧版 git reset HEAD file） $ git reset index.html # 回退到指定提交（⚠️ 丢弃之后所有修改） $ git reset --hard 1a2b3c4
⚠️--hard 会丢失未提交的修改！如果已经 push 到远程，用 git revert 代替。
git revert 撤销
通过创建一个"反提交"来撤销之前的修改（安全！）
📖 语法
git revert 提交ID
💡 示例
# 撤销最近一次提交 $ git revert HEAD # 会创建一个新的提交，内容是对那次提交的反向操作 # 撤销指定范围的提交 $ git revert HEAD~3..HEAD # 不自动提交（方便修改提交信息） $ git revert --no-commit 1a2b3c4
💡多人协作用 revert 而不是 reset！revert 是安全的"反悔"，不影响他人历史。
git clean 清理
删除工作区中未被跟踪的文件和目录
📖 语法
git clean [选项]
💡 示例
# 预览待删除的文件 $ git clean -n # 删除未跟踪的文件 $ git clean -f # 删除未跟踪的目录 $ git clean -fd # 连 .gitignore 里的文件也删掉 $ git clean -xdf
⚠️先 git clean -n 预览，确认无误再加 -f。误删就找不回来了！
git reflog 救命
查看 HEAD 的所有历史移动记录（最后的救命稻草）
📖 语法
git reflog
💡 示例
# 查看所有操作历史 $ git reflog 1a2b3c4 (HEAD -> main) HEAD@{0}: reset: moving to HEAD~1 5e6f7g8 HEAD@{1}: commit: feat: 新增导航栏 9a0b1c2 HEAD@{2}: commit: fix: 首页样式 # 恢复到误操作之前的状态 $ git reset --hard HEAD@{1}
💡git reflog 是后悔药！不管怎么 reset、rebase 搞错了，只要 reflog 里还有记录就能恢复。
git commit --amend 修改
修改最近一次提交的信息或内容
📖 语法
git commit --amend
💡 示例
# 修改上次提交的信息 $ git commit --amend -m "fix: 正确描述这次修改" # 忘记 add 某些文件？补救： $ git add missed-file.js $ git commit --amend --no-edit # 把新文件加入上次提交，不修改提交信息
⚠️只适合修改还没有推送的提交。已 push 的别改——强行推送会害队友！

暂存管理

4 条
git stash 暂存
临时保存工作区的修改，恢复干净的目录
📖 语法
git stash
💡 示例
# 暂存当前修改（恢复到上次提交的状态） $ git stash # 暂存并添加描述 $ git stash push -m "导航栏样式尚未完成" # 暂存未跟踪的文件 $ git stash -u # 列出所有暂存 $ git stash list stash@{0}: On feature-nav: 导航栏样式尚未完成 stash@{1}: On main: 临时修改
💡经典场景：正在改代码，突然需要切换到别的分支修 Bug —— git stash 存起来，修完再取回。
git stash pop 恢复
恢复最近暂存的修改并删除暂存记录
📖 语法
git stash pop # 恢复最近一次暂存 git stash pop stash@{n} # 恢复指定暂存
💡 示例
# 恢复最近的暂存 $ git stash pop # 恢复指定的暂存 $ git stash pop stash@{1}
git stash apply 应用
应用暂存的修改但保留暂存记录（不删除）
💡 示例
# 应用暂存但保留记录 $ git stash apply # 应用后手动删除暂存 $ git stash drop stash@{0} # 查看暂存内容但不应用 $ git stash show -p stash@{0}
💡pop = apply + drop。想同一份暂存应用到多个分支就用 apply。
git stash drop / clear 清理
删除暂存记录
💡 示例
# 删除单个暂存 $ git stash drop stash@{0} # 清空所有暂存（⚠️ 慎用！） $ git stash clear

git diff 差异
查看工作区和暂存区的文件差异
📖 语法
git diff [选项] [文件]
💡 示例
# 查看工作区 vs 暂存区的差异（未暂存的修改） $ git diff # 查看暂存区 vs 上次提交的差异（已暂存的修改） $ git diff --staged # 查看两个分支的差异 $ git diff main..feature # 查看两个提交之间的差异 $ git diff 1a2b3c4..5e6f7g8 # 只看文件名列表（不显示具体内容） $ git diff --name-only
💡看代码改了什么再提交 —— 避免不小心提交了调试代码！
git difftool 可视化
使用外部工具查看差异（比命令行直观）
💡 示例
# 配置使用 vscode 作为 diff 工具 $ git config --global diff.tool vscode $ git config --global difftool.vscode.cmd "code --wait --diff $LOCAL $REMOTE" # 使用 difftool $ git difftool
git blame 追溯
逐行查看文件是谁在什么时候修改的
📖 语法
git blame 文件
💡 示例
# 看文件每行是谁写的 $ git blame index.js 1a2b3c4 (张三 2026-03-15) function login() { 5e6f7g8 (李四 2026-04-20) const token = getToken(); # 只查看特定行的记录 $ git blame -L 10,20 index.js
💡"这行代码是谁写的？" —— git blame 就是干这个的！追责、背锅、学习皆可用。
git show 查看
查看某个提交的详细信息（内容+差异）
💡 示例
# 查看最近一次提交 $ git show # 查看指定提交 $ git show 1a2b3c4 # 查看某个标签对应的提交 $ git show v1.0.0 # 只看提交信息不看代码改动 $ git show --no-patch 1a2b3c4

协作开发

4 条
git rebase 变基
将当前分支的提交移动到另一个分支顶端，使历史更线性干净
📖 语法
git rebase 目标分支
💡 示例
# 把 feature 分支的提交移到 main 顶端 $ git switch feature $ git rebase main # 交互式变基（合并/修改提交历史） $ git rebase -i HEAD~3 # 把最近 3 个提交合为 1 个、修改顺序、更改信息等 # 交互式变基选项： pick 保留提交 squash 合并到上一个提交 reword 修改提交信息 edit 修改提交内容 drop 删除提交
⚠️已经推送到远程的公共分支 不要 rebase！会导致别人的本地和远程历史不一致。
rebase 只适合本地分支或单人开发分支。
git cherry-pick 挑拣
将某个或某些提交复制到当前分支
📖 语法
git cherry-pick 提交ID
💡 示例
# 复制单个提交 $ git cherry-pick 1a2b3c4 # 复制连续多个提交 $ git cherry-pick a1b2c3d..e5f6g7h # 只复制修改但不自动提交（方便修改） $ git cherry-pick -n 1a2b3c4
💡经典场景：hotfix 修好了 main 分支的 Bug，用 cherry-pick 复制到 release/ 分支，不用合并整个 main。
⚔️ 冲突处理 实战
合并冲突时的标准处理流程
📖 冲突标记
# 出现冲突时文件内容长这样： <<<<<<< HEAD（当前分支的版本） console.log("当前分支代码"); ======= console.log("合并进来的代码"); >>>>>>> feature-branch（被合并的分支）
📖 解决流程
# 1. 查看哪些文件冲突 $ git status # 2. 手动编辑文件解决冲突（删掉 <<< === >>> 标记） # 3. 标记已解决 $ git add resolved-file.js # 4. 完成合并 $ git commit
💡git merge --abort 可以取消当前合并，回到开始前的状态。先 abort 冷静一下再重来！
🔄 日常协作流程 流程
多人协作的 Git 标准工作流
📖 一天的工作
# 1. 开始新功能前：先拉取最新代码 $ git switch main $ git pull --rebase # 2. 从最新 main 创建功能分支 $ git switch -c feature-my-work # 3. 开发过程中：多次提交 $ git add -A && git commit -m "wip: 进度1" ... $ git add -A && git commit -m "wip: 进度2" # 4. 推送前：同步一下上游代码避免冲突 $ git fetch origin $ git rebase origin/main # 或 git pull --rebase origin main # 5. 推送功能分支 $ git push -u origin feature-my-work # 6. 去 GitHub/GitLab 创建 Pull Request (PR) # 7. PR 合并到 main 后，本地删除分支 $ git switch main $ git pull $ git branch -d feature-my-work

配置管理

4 条
git config 配置
配置 Git 的全局或项目级设置
📖 语法
git config [--global] 配置项 值
💡 示例
# 设置用户名和邮箱（首次使用必设！） $ git config --global user.name "Chen Zixu" $ git config --global user.email "zixu@example.com" # 推荐有用的全局配置 $ git config --global pull.rebase true # git pull 默认用 rebase 而不是 merge $ git config --global init.defaultBranch main # 初始化仓库默认分支名为 main $ git config --global core.autocrlf input # 修复 Windows / Linux 换行符问题 # 查看所有配置 $ git config --list
💡项目级配置（不加 --global）存在项目的 .git/config 里，覆盖全局设置。
.gitignore 忽略
指定 Git 不需要跟踪的文件和目录
📖 基本语法
# 忽略 node_modules 目录 node_modules/ # 忽略所有 .log 文件 *.log # 但保留 critical.log !critical.log # 忽略 .env 文件（敏感信息！） .env
📖 常见 .gitignore 模板
# Node.js node_modules/ dist/ .env *.log # Python __pycache__/ *.pyc .venv/ *.egg-info/ # macOS / Windows .DS_Store Thumbs.db # IDE 配置 .vscode/ .idea/
💡模板参考：github/gitignore — 不同语言和工具的 .gitignore 示例都有！
git alias 别名
设置 Git 命令别名（提高效率）
💡 示例
# 设置别名 $ git config --global alias.co checkout $ git config --global alias.br branch $ git config --global alias.ci commit $ git config --global alias.st status # 设置复杂别 $ git config --global alias.lg "log --oneline --graph --decorate --all" $ git config --global alias.unstage "reset HEAD --" # 使用别名 $ git lg # ← 等价于一长串 git log $ git ci -m "消息" # ← 等价于 git commit -m
💡推荐配置：git lg 看历史树、git unstage file 取消暂存。
习惯后效率翻倍！

配置 SSH Key 免密连接 GitHub/GitLab
📖 操作步骤
# 1. 生成 SSH 密钥 $ ssh-keygen -t ed25519 -C "your_email@example.com" # 一直回车使用默认位置，可选设置密码 # 2. 查看公钥 $ cat ~/.ssh/id_ed25519.pub # 3. 复制公钥内容，添加到 GitHub → Settings → SSH and GPG keys # 4. 测试连接 $ ssh -T git@github.com Hi username! You've successfully authenticated...
💡配置后 clone 用 SSH 地址：git@github.com:user/repo.git，不用再输密码！

git tag 标签
给某个提交打标签（通常用于版本发布）
📖 语法
git tag [选项] 标签名
💡 示例
# 创建轻量标签 $ git tag v1.0.0 # 创建附注标签（推荐，带作者和日期信息） $ git tag -a v1.0.0 -m "Release v1.0.0 - 登录功能完成" # 给历史提交打标签 $ git tag -a v0.9.0 1a2b3c4 -m "Beta 版" # 列出所有标签 $ git tag # 推送标签到远程 $ git push origin v1.0.0 # 推送所有标签 $ git push --tags
💡版本号规范：语义化版本（SemVer）
v主版本.次版本.补丁 — 如 v2.1.3
git bisect 二分
二分查找 —— 定位 Bug 是哪个提交引入的
📖 使用
# 启动 bisect $ git bisect start # 告诉 Git"这个提交是坏的"（当前 HEAD） $ git bisect bad # 告诉 Git"某个旧版本是好的" $ git bisect good v1.0.0 # Git 会自动切换到中间的提交。 # 测试这个版本，然后标记： $ git bisect good # 如果没问题 $ git bisect bad # 如果有问题 # 重复几次，Git 会精确到引入 Bug 的那个提交 # 结束 bisect $ git bisect reset
💡自动 bisect：git bisect run npm test —— 用测试脚本自动跑！适合回归测试定位问题。
git worktree 多目录
在多个目录同时检出不同的分支
📖 语法
git worktree add 路径 分支名
💡 示例
# 在隔壁目录检出一个分支 $ git worktree add ../project-hotfix hotfix/critical # 列出所有 worktree $ git worktree list # 完成后删除 $ git worktree remove ../project-hotfix
💡再也不用来回 switch 了！main 在 A 目录写代码，feature 在 B 目录同时开发，互不干扰。
git submodule 子模块
在项目中引用其他 Git 仓库（依赖管理）
📖 使用
# 添加子模块 $ git submodule add https://github.com/user/lib.git lib/ # 克隆包含子模块的项目（首次） $ git clone --recurse-submodules https://github.com/user/project.git # 更新子模块到最新 $ git submodule update --remote # 如果忘了 --recurse-submodules $ git submodule update --init --recursive
💡子模块挺麻烦的……能用包管理器（npm/pip/go modules）就用包管理器，
只有在确实需要"把别人的 Git 仓库嵌入自己的项目"时才用。
git hooks 钩子
在特定 Git 操作前后自动执行脚本
📖 常用钩子
pre-commit 提交前运行（检查代码格式、跑 lint）
pre-push 推送前运行（跑测试）
post-merge 合并后运行（安装新依赖）
prepare-msg 编辑提交信息前运行
💡 示例
# .git/hooks/pre-commit 示例（防止提交 console.log） #!/bin/sh if git diff --cached | grep -E "console\.(log|debug)"; then echo "❌ 提交中包含 console.log，请先清理！" exit 1 fi
💡.git/hooks/ 里的钩子不会提交到仓库。团队统一管理的话用 husky（npm）或 pre-commit（Python）。

Git 常用的快捷操作一览
📖 快捷操作
git add -A && git commit -am "msg" # 添加所有文件并提交（快捷但别滥用） git commit --allow-empty -m "start" # 创建空提交（标记起点） git log -S"关键词" # 搜索提交中新增或删除了"关键词"的提交 git diff --word-diff # 按单词对比差异（比默认的行对比更细） git shortlog -sn # 按提交者统计提交次数（看谁最勤奋） git checkout - # 切换到上一个分支（像 cd - 一样方便） git branch -m old-name new-name # 重命名当前分支
💡把这些记在本子上或设成别名，刚开始不用记太多，
最常用的其实就 5 个：add, commit, push, pull, status。
